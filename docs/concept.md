# Yori（より）- サービスコンセプトドキュメント

## サービス概要

**サービス名**: Yori（より）
**サブコピー**: 「寄り添い」の寄り、「より良く」のより

### ひとことで言うと
障害のある子を育てる親が、今日のしんどさをAIに話す。言葉になったら、専門家や支援者に渡せる形になる。

---

## 解決する課題

### ターゲットユーザー
- 障害のある子ども（主に自閉スペクトラム傾向、5〜12歳）を育てる親
- 特に「孤立しがちなパパ」を最初のターゲットに据える

### 課題の本質
- しんどさは言語化できていない、ではなく「**言語化する機会がなかっただけ**」
- 話す場所はある（コミュニティ、ケースワーカー）でも「**一人では踏み出せない**」
- 専門家の存在は知っている、でも「**うまく話せるか不安で繋がれない**」
- 自分のためにケアを受けることへの抵抗がある（特にパパ）

### 今週一番しんどかった瞬間（ユーザーインタビューより）
- 子どもが崩れた後、一人で立て直してそのまま誰にも言えなかった
- きょうだい児を後回しにしてしまった罪悪感
- 配偶者との温度差を感じた夜
- また怒鳴ってしまった直後の自己嫌悪（「またやってしまった」という繰り返し感）

---

## サービスの体験設計

### 入口の設計（重要）
「あなたのために」ではなく**「子どものためにどうすればいいか相談する」** という入口から始める。
話しているうちに、自分のしんどさが自然に出てくる設計にする。

### 体験の流れ

```
0. AIから「今日どうでしたか？」と話しかけてくれる（プッシュ通知）
   ↓
1. 子どものことを話す（入口）
   ↓
2. 話すうちに自分のしんどさが出てくる
   ↓
3. しんどさが言語化・ログ化される
   （例：「次男の発熱と長男が春休みで妻のPMSが重なって地獄でした」）
   ↓
4. 積み重なったログを持って専門家に相談できる
   ↓
5. 相談先をサジェスト（将来的にはマッチング）
```

### なぜAIか
- 評価されない
- 気を遣わなくていい
- 何時でもいい
- 人には話せないことが話せる
- 弱さを見せることへのハードルが低い

---

## MVP機能定義（v1）

### 機能1: AIとの対話（チャット）
**最も重要なコア機能**

- AIが毎日話しかけてくれる（「今日どうでしたか？」）
- 子どものことを話す入口から始まる
- 共感・寄り添いベースの応答
- 判断・評価をしない
- 自然な流れでしんどさが言語化される
- 24時間いつでも使える

**AIのキャラクター設計**
- 温かく、押しつけがましくない
- 問いかけ型（一方的に情報を渡さない）
- 「それはつらかったですね」より「どんな瞬間が一番きつかったですか？」
- 解決策を急がない

### 機能2: 記録・ログ機能
**対話から自動生成される**

- 対話内容から「今日の出来事」「今日の気持ち」を自動抽出・保存
- ユーザーが意識して記録しなくていい（対話するだけで溜まる）
- カレンダー形式で振り返れる
- 「今週しんどかったこと」をサマリー表示

**ログの構造**
```
日付
├── 出来事（事実）: 「次男が発熱、長男が春休みで在宅」
├── 気持ち（感情）: 「地獄だった、誰にも頼れなかった」
└── タグ: [きょうだい, 体調, ワンオペ]
```

### 機能3: 相談先サジェスト（v1では簡易版）
- ログの内容から「今の状態に合った相談先」を提示
- カウンセラー / ケースワーカー / 心療内科 の3分類から候補表示
- 「このログを持って相談してみませんか？」という背中押し
- 将来的にはマッチング機能へ

## 画面設計（必要な画面一覧）

### 1. ホーム画面
- AIからの話しかけ（今日のメッセージ）
- 直近のログサマリー（今週のしんどさスコア的なもの）
- チャットへのCTA

### 2. チャット画面
- メインの対話UI
- シンプルなメッセージ形式
- AIのアイコン（温かみのあるキャラクター）
- 送信後に「今日のログに保存しました」の表示

### 3. ログ一覧画面
- カレンダー or タイムライン形式
- 日ごとの出来事・気持ちのサマリー
- タップで詳細表示

### 4. ログ詳細画面
- その日の対話から抽出された記録
- 「この内容で相談してみる」ボタン

### 5. 相談先サジェスト画面
- 現在の状態の簡易アセスメント
- 相談先の候補リスト
- ログをそのまま持っていける形式（PDF出力など）

### 6. オンボーディング（初回のみ）
- 「子どものことを話す場所です」という入口設定
- 子どもの基本情報（障害種別、年齢）
- 通知設定

## デザイン方針

### トーン
**温かみ・やわらかい**

### カラーパレット（案）
- ベース: クリームホワイト / 温かみのあるオフホワイト
- メイン: くすんだサーモンピンク or モーブ（くすみ紫）
- アクセント: テラコッタ or ウォームブラウン
- テキスト: ダークグレー（黒すぎない）

### タイポグラフィ
- 丸みのあるフォント
- 行間広め
- 文字サイズ大きめ（疲れた目でも読みやすい）

### UIの原則
- 考えなくていい（選択肢を減らす）
- 圧迫感がない（余白を大きく取る）
- 「また来たい」と思える温かさ
- アニメーションは控えめ・やさしい動き

### 避けること
- 医療・福祉っぽい冷たいデザイン
- 情報過多
- 機能を詰め込みすぎたUI
- 明るすぎる・元気すぎる色使い

## 技術スタック（提案）

### フロントエンド
- Flutter（iOS/Android両対応、将来のアプリ化を見据える）
- またはReact Native

- システムプロンプトで「寄り添い・共感ベース」を徹底
- 解決策を急がない
- 問いかけ型の応答スタイル
- ログ抽出用のプロンプトを別途設計（対話終了後に自動実行）

## 成功指標（MVP段階）

- 継続率: 3日後50%、7日後30%
- 対話継続時間: 1セッション平均5分以上
- ログ蓄積数: 1ユーザーあたり週3回以上の対話
- 定性: 「話してよかった」と感じるか（NPS）

## このドキュメントの背景

このコンセプトは、障害のある子どもを育てる当事者の父親との対話から生まれました。

「ビジネスの相談」として始まった会話の中で、当事者自身のしんどさが自然に言語化され、「誰にも話せていなかった」という孤独が明らかになりました。

その体験そのものが、Yoriの体験設計の原型です。

**最初のユーザーは、このサービスを作っている人自身。**

## 認証設計

### 方針
**Googleログイン（OAuth 2.0）のみ**

### 理由
- ボタン一つで完結、フォーム入力不要
- 疲弊した親でも離脱しない
- 「本人しか見れない」という安心感を担保する

### 実装方針
- Firebase Authentication の Google プロバイダを使用（推奨）
- またはNextAuth.js（Next.jsの場合）
- ログイン状態はセッションで管理
- ログアウト機能あり

### データの紐付け
- ユーザーIDに対してチャットログ・記録を紐付けて保存
- 他のユーザーのデータには一切アクセスできない設計

### 画面フロー
```
初回アクセス
↓
ログイン画面（「Googleでログイン」ボタンのみ）
↓
Google認証
↓
ホーム画面
```

### ログイン画面のデザイン方針
- シンプル・ミニマル
- Yoriのロゴ＋一言コピー＋Googleログインボタンのみ
- コピー例：「今日のしんどさを、話してみませんか。」
- 余計な情報を置かない

## 認証設計（追記）

**方式: Googleログイン（Supabase Auth + Google OAuth 2.0）**

### 選定理由
- ボタン一つで完結、登録フォーム不要
- 「今日しんどい」瞬間の離脱を防ぐ
- 「本人しか見れない」という安心感を担保する

### 実装方針
- Supabase AuthのGoogle OAuthを使用
- 未ログイン状態ではチャット・ログに一切アクセスできない
- セッション管理はSupabaseに委譲
- Firebase AuthからSupabase Authに変更（統合しやすいため）

### ランディング画面の設計方針
- 「話す場所です」という一言のみ
- 「Googleでログイン」ボタン1つだけ
- 疲れた人が読む前提なので説明は最小限に

### 画面フロー
```
ランディング（未ログイン）
  ↓「Googleでログイン」ボタン
Google認証
  ↓ 認証成功
ホーム画面
```
